Proceso de migración

 

Este proceso consiste en pasar un sitio definido en WebBuilder 3.2 a SemanticWebBuilder 4.0.

Esto se hará teniendo una instancia de WebBuilder 3.2 en donde radica el sito o sitios que se deseen migrar a la versión SemanticWebBuilder 4.0. Para esto se necesitaría en esta misma instancia prepararla, configurarla para llevar a cabo el proceso.

La forma en que se desarrolló este script, consiste en los elementos que existen en la instancia de WebBuilder 3.2, son tomados de memoria y se crean los elementos correspondientes que serían su equivalente en la nueva versión.  Esto se pensó así porque ya no se podrían copiar los datos de las tablas directamente, ya que la definición de la base de datos es diferente una de la otra. En la versión actual se generan las tablas requeridas para cada sitio.

Esta aplicación o ejecución del script de migración corre en esta misma instancia de WebBuilder 3.2, lo único que se necesita hacer es conocer la ruta destino en donde se migrarán los archivos de los recursos y seguir los pasos de migración.

 

 

 

Requisitos:

 

·         Instancia de WebBuilder 3.2 en donde está el sitio a migrar.

·         Configuración de la base de datos para el sitio de SemanticWebBuilder 4.0

·         Instancia de SemanticWebBuilder 4.0 destino, en donde se migrará el sitio.

 

Preparación y/o configuración de la instancia.

Durante este paso de preparación y/o configuración de la instancia de WebBuilder 3.2 se debe de dar de baja el servicio de  esta misma, ya que se modificarán los archivos de configuración y se agregarán archivos correspondientes de la versión de SemanticWebBuilder 4.0.

 

Configuración de archivos

Para la configuración correcta de la instancia de WebBuilder 3.2 para realizar una migración exitosa del sitio se necesita configurar lo siguiente:

db.properties:

En este archivo se agrega una nueva conexión con el nombre de swb que apunta a la base de datos que se utilizará para la instancia de SemanticWebBuilder 4.0, es muy importante que tenga este nombre.

Ejemplo de configuración de base de datos:

Para una base de datos MySQL

swb.url=jdbc:mysql://localhost:3306/swb_mig

swb.maxconn=50

swb.user=root

swb.password=

swb.idle_time=900

 

web.properties:

En este mismo archivo se agregan las llaves existentes del archivo web.properties de la versión de SemanticWebBuilder 4.0, se tiene que revisar el archivo que estas llaves no se repitan. Al final del archivo se tiene que agregar la llave: swbrep/repositoryTrusted=true, que es utilizada para la creación de contenidos de office en la nueva versión de SemanticWebBuilder 4.0.

 

 

Archivos necesarios

Se necesitan copiar archivos de la versión de SemanticWebBuilder 4.0 a la instancia de WebBuilder 3.2.

Los archivos son:

·         logging.properties: copiar en [wb3]\WEB-INF\classes\

·         MigrateOfficeContents.class: copiar en [wb3]\WEB-INF\classes\org\semanticwb\migration\office\; o si esta en un JAR, copiarlo en [wb3]\WEB-INF\lib\

·         MigrationTool.jsp: script de migración, copiar en [wb3]\

·         Folder de SemanticWebBuilder 4.0 ubicado en [swb4]\WEB-INF\owl, copiarlo a [wb3]\WEB-INF\owl; todo su contenido.

·         Folder de SemanticWebBuilder 4.0 ubicado en [swb4]\WEB-INF\xml, copiarlo a [wb3]\WEB-INF\xml; todo su contenido.

·         Folder de SemanticWebBuilder 4.0 ubicado en [swb4]\WEB-INF\lib, copiarlo a [wb3]\WEB-INF\lib; todo su contenido, revisar si no existen JARs duplicados.

·         Es necesario no duplicar el JAR de Xerces, ya que a veces lo incluye el Tomcat, si este existe en [wb3]\WEB-INF\lib, se tendría que quitar; ya que puede marcar errores al correr el script de migración.

·         En caso de que la versión de WebBuilder 3.2 venga de una migración de WebBuilder 2, posiblemente requiera el WB2Compatibility.jar y WBResourcesComp.jar

Teniendo ya la configuración correcta y haber copiado todos los archivos necesarios, se debe de levantar el servicio del servidor en donde está la instancia de WebBuilder 3.2.

 

Ejecución del proceso de migración

 

Para empezar a ejecutar el proceso de migración, ya teniendo todo preparado y listo. Levantar el servicio de la instancia de WebBuilder 3.2, que puede ser este un Tomcat, Jetty.

Etapa 1:

A continuación se ejecuta el JSP o script de migración a través de un navegador:

http://[localhost]/MigrationTool.jsp

Mostrando la siguiente pantalla:

 

Aquí se debe de poner la ruta destino en donde se copiarán los archivos relacionados con plantillas y con los recursos. Se da enviar y se carga las librerías correspondientes a Semantic WebBuilder 4.0 que servirán para ir creando los elementos correspondientes.

Archivos adicionales como son clases o de otro tipo de archivos, se tendrán que copiar manualmente.

 

Etapa 2:

En este paso se muestra una lista de los repositorios de usuarios existentes definidos en WebBuilder 3.2 y los correspondientes en Semantic WebBuilder 4.0.

En esta etapa muestra la siguiente imagen:

 

Aquí se selecciona el repositorio de usuarios a migrar, mostrando la leyenda de Repositorio disponible, en el segundo recuadro se indica si al crear el repositorio de usuarios, se incluye el copiado de roles y de usuarios existentes en éste. En el último recuadro se muestra la lista de repositorios de usuarios existentes en Semantic WebBuilder 4.0, si se parte de una base de datos vacía para Semantic WebBuilder 4.0, esta aparecerá vacía.

Esta etapa del proceso varía en tiempo, ya que dependerá del número de elementos o registros existentes de roles y de usuarios definidos en WebBuilder 3.2.

Al dar enviar, se creará el repositorio de usuarios a migrar el cual se ligará en el paso siguiente al sitio que se quiera migrar.

Como resultado del proceso se muestra el resumen de el número de usuarios y de roles que se importaron al nuevo repositorio de usuarios.

Etapa 3:

En esta etapa se muestra una lista de los sitios disponibles que se pueden migrar. También se muestra el resumen del resultado del proceso anterior.

 

Al darle Enviar, muestra el número de secciones que forman la estructura del sitio. En esta etapa se genera la estructura del sitio, así como las relaciones entre las secciones si existen secciones virtuales, así como la asociación del repositorio de usuarios seleccionado en la etapa anterior.

 

El tiempo del proceso de creación e importación de la estructura del sitio depende del número de secciones que lo componen. Esta duración puede ser de minutos hasta horas, hay que ser pacientes. Depende del volumen de información que compone el sitio. Existe la opción de regresar y seleccionar algún otro sitio disponible para migrar.

 

Etapa 4:

En esta etapa del proceso muestra el resultado del proceso de la importación de la estructura del sitio.  Indicando el número de secciones importadas, así como el número de relaciones que se establecieron entre las secciones.

Esta etapa cuenta con tres pasos para importar los catálogos y/o la información que tiene el sitio seleccionado. Está separado de esta forma por la dependencia que existe entre el tipo de elementos.

 

Paso 1:

 A continuación se muestra la lista de catálogos que se pueden importar de WebBuilder 3.2, inicialmente están seleccionados todos.  Aquí se importan lenguajes, dispositivos, Dns, reglas, flujos de publicación, tipos de recursos y grupos de plantillas.

En los recuadros subsecuentes, muestra lo que sigue a importar pero no se pueden seleccionar hasta que se importen las dependencias correspondientes.

 

Como resultado de este proceso se activarán los catálogos que se pueden importar en el siguiente recuadro.

Paso 2:

Continuando con la importación de catálogos,  en el recuadro del Paso 1 de 3, se muestra el número de elementos creados de cada uno. Hay dependencias entre los pasos de importación de catálogos. No se pueden importar Sub-Tipos de recursos si en el paso anterior no se importaron Tipos de recursos, y tampoco se podrá importar las Plantillas, si en el paso anterior no importaron Grupos de plantillas.

 

 

En este Paso 2 de 3 se pueden seleccionar los Sub-Tipos de recursos y/o Plantillas.

El proceso que se llevará a cabo es agregar cada sub tipo de recurso al tipo de recurso importado si es que existe alguna dependencia, no necesariamente debe de existir algún sub-tipo de recurso; mientras que en las plantillas, se asocian al grupo de plantilla importado y al procesamiento y copiado de los archivos que contengan cada plantilla. Si se tienen muchas versiones en las plantillas, se tomará la versión actual y sólo se copiará esa versión dejando  en SemanticWebBuilder 4.0 una única versión de la plantilla copiada. Incluye la corrección de las rutas definidas dentro de las plantillas, en relación a la ruta que se recibió como parámetro inicialmente.

 

 

Paso 3:

Continuando con la importación de catálogos,  en el recuadro del Paso 2 de 3, se muestra el número de elementos creados de cada uno. Hay dependencias entre los pasos de importación de catálogos. No se pueden importar Recursos si en el paso anterior no se importaron Sub-Tipos de recursos, si es que existiera alguna dependencia al Sub-Tipo.

 

 

En este Paso 3 de 3 se pueden seleccionar únicamente Recursos, que son los elementos faltantes de la importación del sitio.

 El proceso que se llevará a cabo es agregar cada recurso, verificar su calendarización, su filtro, revisa también si tiene asociado alguna regla o algún rol, así como si es un contenido y con esto asociarlo a la sección en donde se debe de dar de alta y de esta forma quede asociado con cada uno de los elementos involucrados.

Durante este proceso se copian archivos relacionados a los recursos, colocándolos en la ruta correspondiente y si aplica, se corrigen rutas que estén definidas en estos.

El tiempo de ejecución depende del volumen de información que se tenga, puede tardar de varios minutos hasta horas. Si el recurso cuenta con versiones, como el caso de LocalContent, reinicia las versiones y deja únicamente la versión actual, dejándola como versión inicial.

Terminando la migración:

El resultado de este proceso Paso 3 de 3, muestra el total de los recursos importados.

Si se requiere importar otro sitio, se da Terminar y vuelve a empezar este mismo proceso.

 

 

Pruebas del Resultado del proceso de migración

Finalmente para probar el sitio migrado, se deberá terminar, o dar de baja el tomcat, ya que se tienen librerías de SemanticWebBuilder 4.0 utilizando la base de datos que se requiere para levantar el servicio de Tomcat para la instancia de SemanticWebBuilder 4.0

Levantando el servicio

Levantar el servicio del servidor de aplicaciones en donde se encuentra configurada la instancia de SemanticWebBuilder 4.0 que debe de utilizar la base de datos utilizada en el db.properties swb.

Terminando de levantar el servicio, entrar a la administración de la herramienta SemanticWebBuilder 4.0, http://[localhost:8080]/swbadmin la pantalla deberá de incluir en la lista de sitios del lado izquierdo, uno con el nombre del sitio migrado, en este caso llamado Prueba migración, en este elemento se puede revisar si todos los elementos relacionados al sitio se migraron correctamente.
